## 2026-05-19 — pytest orphan ate 16 GB; mutation-resistant test design

**Trigger**: user reported the laptop was running hot. Diagnosis found
PID 34109 — an orphaned `pytest tests/unit/test_cache_keepalive.py::TestRealEventBeatsKeepalive -q` —
holding 16 GB RSS, 47-59% CPU, state `stuck`, runtime 27:35, PPID=1.
The host had 31/32 GB used, 13 GB compressor, 1.3 M swap-outs. Killing
the orphan immediately freed ~6 GB and the compressor pressure dropped.

**Root cause** (three bugs compounded — single-fault tolerance held, two-fault
tolerance did not):

1. **`MagicMock.call_args_list` is unbounded.** The test patched
   `httpx.get` with `MagicMock(return_value=...)`. `await_events` is a
   `while True:` loop that calls `httpx.get` ~80k times per second when
   the loop body never returns. `MagicMock` records every call — measured
   ~500 MB/s allocation rate; macOS compressor saturates around 16 GB and
   transitions the process to `stuck`. So a logic bug that *should* have
   surfaced as a hang surfaces as a host-OOM instead.
2. **`TestRealEventBeatsKeepalive` had no internal bound.** The test
   relied on the prior session's mutation harness using
   `subprocess.run(timeout=60)` to kill a hung pytest. Run as plain
   `pytest -q` (no `--timeout=`, no `pytest-timeout` in deps), nothing
   reins it in. The dev-log entry from the prior session even flagged
   this as fragile (line 176: "consider an explicit max-poll-count
   assertion"); the warning was not acted on.
3. **`subprocess.run(timeout=60)` only kills the direct child.** `uv run
   pytest` produces a process tree where `uv` is the direct child and
   `pytest` is a grandchild. When `subprocess.run` timed out it killed
   `uv`; `pytest` re-parented to PID 1 and continued. This is the
   orphaning mechanism. The mutation harness in session
   `a72f9819...` (2026-05-18 16:54) was the original launcher — the
   orphan survived for ~33 hours before user noticed the heat.

**Diff-invisible findings**:

- The previous session already saw the brittleness and wrote it into
  the dev-log Open list — "the test catches priority inversion only
  because the loop hangs and pytest's 60s subprocess timeout fires."
  No `--timeout` is configured in `pyproject.toml`; pytest-timeout
  isn't in deps. The Open item was correct and load-bearing; ignored.
- The OOM amplification factor (~500 MB/s) means *any* `while True:`
  pattern under a `MagicMock(return_value=...)` patch is a latent host
  killer if the mocked function is high-frequency. Worth scanning the
  rest of the suite for the same shape, not done in this session.
- The actual `await_events.py` implementation is **correct** — `if
  events:` does prioritize real events. There was no production bug.
  The bug was entirely in the test's robustness model.

**Decisions**:

- Replaced the unbounded `MagicMock(return_value=...)` patch with an
  explicit `side_effect` callable that counts polls and raises
  `AssertionError` after `max_polls=4`. Fast-fail under any future
  priority-inversion mutation: 0.10s instead of forever. Verified by
  re-running the same `if False and events:` mutation — it now fails
  with `"keepalive is preempting actual work"` in 0.54s wall-clock,
  with no detectable RSS growth.
- Added an additional positive assertion: `poll_count["n"] == 1`. The
  real event must be returned on the *first* poll, not the fourth.
  This catches subtler regressions where keepalive is delayed but
  still wins eventually.
- Did NOT add `pytest-timeout` to deps. Wall-clock timeouts are a
  defense-in-depth backstop, not a substitute for bounded test
  fixtures. A test that needs an outer timeout to terminate is itself
  buggy. The right discipline is: every loop-driving test must have an
  internal poll/iteration cap that fails with a meaningful assertion.
- Did NOT widen this to a generic "no MagicMock with return_value
  inside loops" lint rule. The pattern is fine when the loop is
  externally bounded; only the *combination* (unbounded loop + unbounded
  mock recording) is dangerous. A linter would have too many false
  positives. The disciplined fix is per-test poll caps.

**Rejected approaches**:

- Adding `pytest-timeout` and a global 30s cap. Would have hidden the
  failure under a generic timeout signal instead of the meaningful
  assertion. The mutation testing story (test must fail loudly under
  priority inversion) is exactly what you lose with a wall-clock cap.
- Changing `await_events()` to add a max-iteration safety guard. The
  production code is correct — adding defensive iteration limits there
  would be the wrong layer. The test is the right layer to bound.
- Hunting for the original orphan-launching session and trying to
  prevent that pattern globally. The mutation-test pattern is useful
  and worth keeping; the lesson is "use `Popen` + `process.kill()` +
  `os.killpg`" if you ever bring back subprocess-driven mutation, not
  "stop doing mutation testing." Not done in this session — no live
  mutation harness in the repo right now.

**Open**:

- Sweep the rest of `tests/` for `while True:` (or near-infinite) loops
  patched by `MagicMock(return_value=...)`. Any other site has the same
  latent OOM. Not done; queue.
- Consider adding to the Role-prompt or contributing guide: "any test

**Trigger**: user asked whether the 5-min TTL cache-keepalive machinery (`wait_bg`, `mos_await_events` synthetic `cache_keepalive` event, `bg_keepalive_nudge` hook, `cache_stats` post-hoc analyzer) could become closed-loop — measure real hit rate, dynamically adjust within a range, and stop polling entirely when the endpoint clearly does not honor long-TTL caching, with periodic recheck.

No code, hooks, MCP tools, skill, or memory changes shipped this session. This entry exists so a future session can pick it up cold instead of re-deriving the analysis.

**Diff-invisible findings** (worth keeping even though we are not building):
- The signal needed for closed-loop control already exists. `~/.claude/projects/<slug>/<id>.jsonl` records `cache_read_input_tokens` / `cache_creation_input_tokens` / `ephemeral_5m_input_tokens` / `ephemeral_1h_input_tokens` per assistant turn; `cache_stats.py` reads exactly these fields. Closing the loop = letting `wait_bg` (and `cache_keepalive_seconds`) consult the same data online instead of only post-hoc.
- Three independent open-loop points, each with its own injection site and decision authority:
  1. `wait_bg(deadline_seconds=180)` in keepalive MCP — main-session bg-task path.
  2. `mos_await_events` / `mos_noter_wait` synthetic `cache_keepalive` event — Role long-session path, currently driven by static `gru.yaml: cache_keepalive_seconds`.
  3. `cache_stats` post-hoc CLI — already produces a recommendation but cannot influence runtime.
  A shared "read last N turns of jsonl → bucket the hit rate" pure function could power all three, but a shared *controller* over-couples three different process boundaries. Share the probe, not the controller.
- **Failure modes that would bite a naive implementation:**
  - `wait_bg` is stdio MCP; it does NOT know its caller's session id. Either caller passes session_id/cwd in, or we scan `~/.claude/projects/` for the freshest jsonl — the latter is unreliable under concurrent sessions. Hard prerequisite before any closed-loop control can run.
  - "cache miss" ≠ "keepalive failed". Prefix changes (new tool def, new cwd, post-`/compact`) legitimately force `cache_create > 0` on the next turn. A naive 1–2 turn window mis-classifies legitimate prefix changes as "keepalive broken" and shuts off heartbeats — exactly the wrong thing. Need a sliding window AND a cheap proxy for "system prompt unchanged" (closest available: ephemeral_5m_creation ≈ system prompt size).
  - Cost asymmetry strongly favors over-warming. One keepalive ≈ 6¢ of `cache_read`; one cold start ≈ 50k+ uncached input tokens (≈ several dollars + visible UX delay). "Stop polling" must require multiple consecutive cold windows, never a single one.
  - **`wait_bg`'s tick payload MUST stay byte-stable** — that is the entire reason it works. Any dynamic field (`recent_hit_rate=0.83` etc.) would break the very cache it is trying to preserve. Health signals must ride channels that do not enter the next turn's prefix: caller-echo subobject, `systemMessage`, separate probe tool. Easy to get wrong on the first try.

**Rejected approaches** (so a future session does not re-derive them):
- "Inject hit_rate into the wait_bg tick" — self-defeating, see byte-stable note above.
- "Read the jsonl every wait_bg call" — race with Claude Code's writer; expensive at high frequency. Out-of-process probe with debounced cache required if pursued.
- "One closed-loop component shared across the three injection sites" — appealing but over-coupling. Share a pure-function probe; keep three controllers.
- "Build endpoint-detection (Vertex / Bedrock / direct anthropic) for an `endpoint_unsupported` bucket now" — would have been the highest-leverage single output (skip ALL keepalives where `cache_control` / 1h beta is ignored), but requires real telemetry from each endpoint. Not worth speculating without data.

**Open** (← future-session prompts, only if the problem actually shows up):
- Trigger to revisit: a real `cache_stats` rollup on Vertex / Bedrock / non-direct endpoint shows `ephemeral_5m_creation` dominating with low overall hit rate while `cache_keepalive_seconds > 0` is firing. That is the concrete signal that blind keepalive is wasting tokens and adaptive control would pay off.
- If revisited, do it in three independent phases — each phase ships value alone:
  1. Add a read-only `cache_health_probe()` MCP tool returning `{recent_turns, hit_rate, ephemeral_1h_share, prefix_changed_recently, recommendation_bucket ∈ {healthy, borderline, cold, endpoint_unsupported}}`. No behavior change; just expose the oracle. Cheap, self-contained, useful even alone.

## 2026-05-19 (night) — Prompt cache invalidation full audit: binary reverse-engineering + online research

**Trigger**: user asked to find ALL behaviors during a Role agent's normal
operation that could cause prompt cache loss, using real evidence (binary
analysis, docs, community reports) rather than training-data reasoning alone.

**Diff-invisible findings**:

- **Claude Code 2.1.143 `xF3` function** (reverse-engineered from binary):
  places cache breakpoints on the last 1–2 non-api_system messages. Second
  breakpoint gated by `tengu_basalt_spur` feature flag via `LoH()`. System
  prompt breakpoints placed by `fc8` → `uF3` with `cacheScope: "org"` or
  `"global"`. The billing header (`x-anthropic-billing-header`) gets
  `cacheScope: null` (never cached).
- **`CLAUDE_CODE_ATTRIBUTION_HEADER=0`** (already set in project
  `.claude/settings.json`) fully disables `Fr_()` which generates the
  `cch=00000` billing header. This prevents the Issue #40652 "cch= hash
  poisoning" attack vector entirely — no billing header = no hash to collide
  with tool results. Confirmed: no independent find-and-replace of `cch=`
  exists in the binary; the vulnerability only manifests when the header is
  present in the system prompt AND tool results echo it back.
- **MCP reconnection does NOT invalidate cache** under normal conditions.
  `PJ6` (tool schema serializer) caches results in an in-memory `Map` keyed
  by `name + inputSchema hash`. Cache is only cleared on OAuth token save /
  logout (`Pn_()`), not on MCP reconnect. As long as the MCP server returns
  the same tool names and input_schemas after reconnect, the API request's
  tool definitions stay byte-identical. The risk is only if tool COUNT
  changes (server partially starts, returns fewer tools, then fully starts
  and returns all tools → two consecutive cache invalidations).
- **Tool schema cache (`oj9` Map) does NOT include description in its key.**
  If a tool's description changes on reconnect but name+input_schema stay
  the same, the OLD cached description is used. This is actually protective
  for cache stability (description flicker from MCP reconnect is absorbed).
- **`ENABLE_TOOL_SEARCH=false`** (set in role env since the e2e session) means
  all tools are eagerly loaded. No deferred-tool stubs, no ToolSearch-driven
  schema appending mid-session. This eliminates the "adding tools mid-session
  invalidates cache" vector for Role agents entirely.
- **Extended thinking block stripping** is unavoidable on Opus 4.7 (default
  model for EACN roles). Each time old thinking blocks are cleared from
  context, the messages prefix changes → messages cache miss. System/tools
  cache is unaffected. This is a per-turn cost that grows with conversation
  length until compact fires.
- **Compaction preserves system/tools cache** (confirmed from `ot7` function
  in binary: `tengu_compact_cache_prefix` flag, uses `DL` fork with same
  system prompt). Only messages cache is rebuilt post-compact.
- **Online research** (background agent, 15 sources): confirmed all findings
  above plus documented the 20-block lookback window, 1024-token minimum
  cacheable prefix, concurrent request race condition, and the March 2026

- `cch=` poisoning: **already mitigated** by `CLAUDE_CODE_ATTRIBUTION_HEADER=0`.
