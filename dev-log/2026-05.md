## 2026-05-19 — scaffold + extension-point audit + red-team + fixes

**Trigger**: user asked me to use the `project-bootstrap` skill to build a
dev scaffold for MinionsOS and use it to check feature consistency. Then
asked me to red-team the scaffold and patch every finding.

**Diff-invisible findings**:

- `minions/lifecycle/skills.py:list_skills` is intact but **has no live
  caller** in this repo — the historical `invoke_role_ephemeral` was
  replaced by the resident-Role model and never re-wired. This means new
  role skills are NOT auto-injected into the Role's system prompt; they
  exist on disk and the Role's SYSTEM.md tells the Role to discover them
  via `list` + `read`. So adding a skill is **cache-safe** (does not
  invalidate any system prompt). The user explicitly worried about this
  cache impact — confirmed false alarm under current architecture.
- The system-prompt cache failure mode that DOES exist: any edit to
  `roles/SYSTEM.md` (common contract), `roles/<role>/SYSTEM.md`, or
  `lifecycle/role.py:_BOUNDARY_TEXT` invalidates that role's
  `--append-system-prompt` block. This is the real blast radius the user
  is right to track. Pending work: a `mos audit` check that hashes these
  files against a baseline; deferred mid-session, see Open below.
- `expert.startswith("expert")` was buggy in 5 places (config / role /
  role_launcher / agent_registry / skills) — `"expertise"` would have
  silently inherited the full `expert` whitelist. Fixed to require exact
  `expert` or `expert-` prefix. The bug was discovered by red-team; it
  is unlikely to have been exploited (no role named `"expertise"`
  exists) but the fix is one-line everywhere and worth doing.

**Decisions**:

- `R5b` (role identity spoofing in `mos_publish_to_shared`) was the
  single most severe finding — fixed via one-line check
  `_enforce_caller_identity` in `mcp_server.py`. Pattern reusable for
  any future MCP tool that takes a free-form `role` argument: check
  against `MINIONS_ROLE_NAME` env. Document this in any new tool spec.
- `R3` (.mcp.json tampering) was solved via `write_boundary_guard.py`
  PreToolUse hook + an explicit `_TRUST_ROOTS` list, not via chmod
  immutable. Reason: macOS immutable flag needs `sudo` and would break
  `install.sh` idempotency. Trade-off accepted: the hook can be bypassed
  by `MINIONS_DISABLE_WRITE_BOUNDARY=1`, which is fine for system
  maintenance scope but means the protection is policy-level not
  filesystem-level.
- `R2` (cross-project port arg) and `R5b` use the same env-vs-arg
  pattern — both raise PermissionError when the supplied arg disagrees
  with the role process's environment. Future MCP tools that take a
  port should use `_enforce_caller_project` for free.
- `Ethics keeps codex` even though it has no Bash/Write/Edit. This was
  flagged by the privilege red-team (`P2`) but reflects the deliberate
  read-mostly-with-codex-delegation design. Documented as exception in
  `audit.py:_CODEX_INTENTIONAL_RESTRICTED_ROLES` rather than being
  flagged each push.
- `Noter / Ethics subagent broader than main` (Write / Edit not in main
  whitelist) is the operating model — main is observer, subagents do
  the writing. Documented in `audit.py:_SUBAGENT_BROADER_EXCEPTIONS`.

**Open** (← future-session prompts):

- **SYSTEM.md drift audit was started, not finished.** Goal: hash every
  SYSTEM.md + `_BOUNDARY_TEXT` and warn at `mos audit` time when they
  change, because every change cold-starts every Role's prompt cache.
  Strict requirement (per user): the check must live entirely in
  `mos audit`, never in a hook, never in a Role-facing surface, never
  in EACN. The user is concerned about audit machinery contaminating
  the autonomous research loop and made this a hard line.
- `write_boundary_guard.py` parses Bash with regex (`_GIT_WRITE_VERBS`)
  and `cd` / `git -C` heuristics. Complex shell — heredocs, nested
  command substitutions, `bash -c "..."` wrappers — could in principle
  evade the parser. Not a worry for ordinary Role code, but if a Role
  ever runs adversarial input through Bash this becomes a hole. Future
  hardening: shell parser instead of regex, or move enforcement into a
  setuid helper.
- `mos audit` is now in the push gate (`/minionsos-push` Step 1 + Step
  6). When push starts using it, watch for false-positive warnings
  becoming noise. The user explicitly wants us to track which warnings

**Rejected approaches**:

## 2026-05-19 01:09 — Cache keepalive infrastructure + Codex-as-subagent dispatch model

**Trigger**: User wanted main session's 5-min prompt cache to survive bg tasks (Bash run_in_background, Agent subagents, codex MCP calls), and asked "Codex MCP can run in background like Claude Code subagent?" → led to wrapping codex in a Haiku Agent + reframing the model tiering.

- **Default deadline tightened 240s → 180s** after one observed cache-cliff event in a 12-min e2e test (240s sleep + tool roundtrip + intermediate TaskOutput pushed inter-turn gap past 300s). Hard ceiling also 300s → 270s defensively.
- **Two-tier subagent dispatch model** in `~/.claude/CLAUDE.md` + `/codex` skill:

  - Tier 2: **Codex GPT-5.5 xhigh as the actor**, wrapped in `Agent(model:"haiku")` as plumbing — Haiku's only job is mechanical args forwarding + structured field copy-back. Sonnet is degraded fallback only.
- **Reframed** "Haiku-relay → Codex" → "Codex (actor) wrapped in thin Haiku Agent (plumbing)". Concept is: we're upgrading codex MCP from synchronous tool to first-class Claude Code subagent via the Haiku wrapper.

**Diff-invisible findings**:
- `--disallowed-tools` is **session-global**: subagent inherits parent's disallow list, no asymmetry possible. Confirmed by direct e2e test (main blocked, subagent also blocked). This means "main forbidden, subagent allowed" architecture cannot be hard-enforced via Claude Code's native permission model.
- MinionsOS `_WHITELIST['subagent']` entries are **prompt-side documentation only**, never enforced — `role_launcher.py:162` only passes the `main` whitelist as `--allowed-tools`. Agent subagents inherit parent permissions regardless of what `_WHITELIST['subagent']` says.
- Soft enforcement via skill + CLAUDE.md works empirically: 2 fresh `claude --print` sessions both auto-followed Tier 2 (haiku-relay-codex) and bg-mode for long task without hard enforcement. Sufficient for now.
- tok.fan relay strips `cache_creation` sub-bucket fields on some response paths — usage observability is sometimes degraded to `input_tokens` only. Not a code bug; a relay quirk.

**Rejected approaches**:
- **Hard-enforce via `--disallowed-tools`** — rejected after e2e test showed subagent also blocked. Documented in memory + skill.
- **Bash wrapper around codex CLI (option B from earlier discussion)** — abandoned in favor of Haiku-Agent wrapper. Same backgrounding effect but Haiku wrapper plays cleaner with Claude Code's first-class subagent infrastructure (run_in_background, TaskOutput, automatic hook nudge).
- **Modify codex-subagent MCP server (option C)** — deferred. Would be cleanest but ~150 LoC TypeScript change with fork/zombie/cleanup concerns. Soft-discipline path is good enough.

**Open**:

- `bg_keepalive_nudge` hook does not yet handle MCP tool calls (e.g. long codex MCP if invoked directly). Currently only Bash|Agent|Task. If anyone calls codex MCP directly with a long task, no hook fires. Caller-side discipline is the only protection.

## 2026-05-19 — cache-cliff env vars guaranteed across dev / Gru / Role

**Trigger**: user noticed `BASH_*_TIMEOUT_MS` / `CLAUDE_AUTO_BACKGROUND_TASKS`
were live in `~/.claude/settings.json` but NOT explicitly guaranteed for
the three MinionsOS agent surfaces. Works on this machine, would silently
break on a fresh checkout. Asked to make all three explicit, mirrored
inside the repo, without touching `~/.claude/`.

**Diff-invisible findings**:

- Three surfaces, three different startup paths, no shared source. Dev-
  host reads `MinionsOS/.claude/settings.json` (project-settings
  discovery); Role goes through `_role_env()` in `role_launcher.py`
  (tmux session env, authoritative); Gru is exec'd directly by
  `bin/gru` (bash script env). The 3-way mirror is the pragmatic
  tradeoff — unifying would require restructuring all three.
- Why Role can't inherit from `MinionsOS/.claude/settings.json`: Role
  cwd is `project_{port}/branches/<role>/`, a git worktree under a
  per-project bare repo. Claude Code's project-settings discovery
  walks up from cwd to that worktree's git root and stops — never
  reaches `MinionsOS/.claude/`. So Role MUST be served via
  `_role_env()`.
- `MinionsOS/.claude/settings.json` is git-tracked (`.gitignore` only
  excludes `*.local.md`). Dev-host fix is therefore already portable;
  only Role and Gru needed new code this session.
- Hook parity verified: `bg_keepalive_nudge.py` byte-identical between
  `~/.claude/hooks/` and `minions/hooks/`. `large_file_guard.py` and
  `edit_failure_rescue.py` differ by one whitespace line each. The
  load-bearing keepalive hook is fully mirrored.

**Decisions**:

- Gru uses `: "${VAR:=default}"` + `export` (operator override wins);
  Role uses literal authoritative values in `_role_env()` (Role env
  is contract, not default). Different on purpose — Gru is operator-
  facing, Role is contract-facing.
- Test pins exact values AND asserts
  `BASH_MAX_TIMEOUT_MS < 5*60*1000` as a hard invariant. Future raise
  above the 5-min cliff becomes a deliberate test diff, not silent
  drift.
- `auto_title_*` hooks deliberately NOT ported. Session-title
  cosmetics; porting would mean Role processes write to user-level
  state, contaminating the autonomous-research loop. Hard line: only
  cache-TTL-relevant hooks come into the project.

**Open** (← future-session prompts):

- The 3-way mirror is now load-bearing but only the Role surface is
  test-pinned. Dev-host (`.claude/settings.json`) and Gru
  (`bin/gru`) values can drift unnoticed. Future hardening (deferred
  until first drift incident): a `mos audit` check that reads
  BASH_DEFAULT_TIMEOUT_MS / BASH_MAX_TIMEOUT_MS /
  CLAUDE_AUTO_BACKGROUND_TASKS from all three sources and warns on
  mismatch. Insertion point: `minions/scaffold/audit.py` next to the
  existing `_ALL_CHECKS` tuple.
- `minions/hooks/bg_keepalive_nudge.py` arrived in this branch with a
