一句话总结

EACN3 是一个 任务市场 + 信誉 + 经济 的去中心化多智能体协议：Python 后端 + TypeScript MCP 插件（47 个 eacn3_* 工具）。 它推荐"一个 server、一个 agent、用 eacn3_next 驱动主循环"的 agent-centric 用法。

MinionsOS 把它适配成 project-centric：每个项目一个 server（按端口隔离），每个 Role 一个 agent，主循环走我们自己的 MOS Agent Pool（mos_await_events + pending.jsonl ACK 补丁），把 EACN3 的 drain-on-read 与 60 秒长轮询硬上限 对 Agent 透明化。适配整体正确且最小化，但未充分利用几个 EACN3 原语： team_setup（天然契合我们的 branch-per-role 布局）、reputation/economy 的可观测性、invite_agent 的定向邀请、close_task 的清理、confirm_budget 的角色间议价。

唯一的多节点缺口：_post_message_raw 发送时 server_id 与 network_id 置空。在当前"每项目单节点"的前提下无害；若未来跨节点联邦，需要修。

1. EACN3 是什么

EACN3（Emergent Agent Collaboration Network）是一个三层协议栈的协调层：

层级协议职责
CoordinationEACN3竞标、仲裁、信誉、发现 —— 决定"谁干活、干得怎样、下次信谁"
CommunicationA2AAgent 之间的消息与会话建立
ToolingMCP工具调用标准

核心原语

  1. 任务市场。Agent 发布任务（budget、domains、deadline），其他 Agent 竞标（confidence、price）。准入公式 confidence × reputation ≥ threshold，由服务端判断。发起者挑选获胜结果，escrow 结算。
  2. 分层分解。中标者可 create_subtask，预算从父任务 escrow 中划拨，depth 上限 3，结果通过 subtask_completed 事件自动冒泡。
  3. Agent-to-Agent 直信（POST /api/messages）。与任务事件共用每 Agent 队列。
  4. 团队形成 (team_setup)：基于共享 git 仓库，带 ACK-tracked 的 per-agent 分支握手。
  5. 信誉系统：0.0–1.0 分，新 Agent 默认 0.5，由 task_completed / task_rejected / task_timeout / bid_declined 驱动。
  6. 经济账户：内部 credit ledger，每 Agent 有 available + frozen（escrow 冻结）。
  7. 发现：servers / agents / domains，Gossip → DHT → Bootstrap 三级查询（多集群场景）。
  8. Cluster 模式：联邦部署；单机部署（MinionsOS 选型）下大部分 cluster 代码是本地短路。

交互模型

4 个顶点，6 条边，全连通：Platform（EACN3）、Human、Agent A、Agent B。MinionsOS 的每个项目相当于一个独立的小宇宙（独立 EACN3 端口），Gru 同时扮演 Human 与 Platform 的联络点。

5. 关键协议不变量

🔑 GET /api/events/{agent_id} 是 drain-on-read

消息返回给 Agent 后即从 OfflineStore 移除。没有 ACK，没有重放。这是 MOS Agent Pool 存在的根本原因。

🔑 timeout 查询参数硬上限 60 秒

routes.py:641 的 Query(default=0, ge=0, le=60) 会 422 拒绝任何 > 60 的值。MOS Agent Pool 在 Python 层分片成 60 秒块来规避。

⏱ Liveness = fetch 活动

poll_events 调用时会 touch_agent_fetch(agent_id)。后台扫描器每 30 秒（scan_interval_seconds）检查一次，120 秒无 fetch 即标记下线（agent_offline_seconds=120）。带 x-server-id 头的 POST/PUT/DELETE 也会刷新 server 上所有 agent 的活性。

🚀 启动门

lifespan 启动未完成前，除 /health 外所有请求返回 503。

🌐 跨节点转发时才需要三层地址

server_id / network_id 只在 cluster 转发（forward_push）时被使用；本地单机投递走 _store.store(target_agent_id, ...)，只认 agent_id。MinionsOS 发空字符串在单机下完全 OK。

🔒 x-server-id 是隐式认证令牌

活性刷新与 admin 端点依赖它。network-client.ts 自动注入；MinionsOS Python 客户端目前不显式注入，但我们的轮询路径通过 touch_agent_fetch 覆盖活性。

📨 V5 使用 HTTP 轮询而非 WebSocket

app.py 注释明确："Every event is written to the per-agent message queue unconditionally. Agents drain it via HTTP polling. No WebSocket — just a queue."

6. EACN3 推荐用法

来自 AGENT_GUIDE.md 与 README.md：

起步 5 步

1. eacn3_health()                                验证节点
2. eacn3_connect(network_endpoint?)              获取 server_id
3. eacn3_register_agent(name, description, ...)  获取 agent_id
4. eacn3_get_events() / eacn3_list_open_tasks()  初始态
5. eacn3_next()                                  驱动主循环

主循环：eacn3_next

每次调用返回一条最高优先级事件，附带 suggested_tool 与 suggested_params。空闲时返回上下文感知的反思 prompt（未取回的委派结果、未回复消息、等等）。EACN3 设计者把它定位为核心调度器。

其他要点

  • 每 server 一个 agent（简化 agent_id / initiator_id / sender_id 的自动注入）。
  • Domains 尽量具体："python-coding" 比 "coding" 路由更精。
  • Budget / Deposit 纪律：available ≥ budget 方能发布任务；deposit 充值。
  • Reputation 由结果驱动：submit_result 自动上报 task_completed；reject_task 自动上报 task_rejected。
  • 禁止手搓 HTTP：AGENT_GUIDE 有专门的 "NEVER bypass" 章节。

7. MinionsOS 哪里对齐 MATCH

  • 每项目一 server、每 Role 一 agent。通过端口隔离 server，EACN3_STATE_DIR 给每个 Role 独立插件状态目录。
  • mos_await_events 对 Agent 透明化 60 秒切片。Agent 眼里是一次调用；Python 层分块轮询。这是在协议硬上限下能给出的最佳方案。
  • Per-wake ACK 补丁。.minionsos/inbox/pending.jsonl 在事件交给 Agent 前本地落盘，崩溃重启后下次 wake 注入 init prompt。EACN3 自身没 ACK，这个缺口正是 MOS Agent Pool 存在的理由。
  • 活性通过 fetch 自带刷新。每 60 秒一次 poll_events，agent 始终 online；休眠 role 超过 120 秒自动离线也合理（MinionsOS 视 Role 为 ephemeral，下次 wake 重新注册）。
  • 非破坏性读直调。eacn3_get_task / eacn3_list_* / eacn3_get_messages 等直调；drain 类工具全部走 MOS。
  • Subagent EACN 不可见。minions-role 代理 profile 正确地从 subagent 白名单剥离所有 eacn3_*。
  • Budget 约定。内部任务 budget=0；ensure_balance 把 Gru 余额顶到 local_eacn_initial_balance（默认 10 000）作保险。

8. MinionsOS 哪里发散 GAP

  1. 完全不用 eacn3_next。 EACN3 设计核心是 next（"core work scheduler"，返回一条事件 + suggested action + urgency）。我们用自己的 mos_await_events 拉原始事件，由 role 提示词里的 Plan → Dispatch → Verify 决定动作。 这是有意的（我们要显式控制流，而不是 EACN3 的每事件自动建议），但代价是丢失了：
    • EACN3 的 urgency 排序（task_broadcast > discussion_update > task_timeout）；
    • 空闲反思 prompt（"你委派了 1 个任务，结果查了吗？"）；
    • server.ts:1835+ 那张久经考验的事件→动作映射表。
  2. 不用 eacn3_team_setup。 EACN3 的 "共享 git repo、每 Agent 一条分支" 团队原语跟我们的 branches/<role>/ 布局几乎完美契合。现在完全未连通，导致：
    • Role 之间无 ACK-tracked 的分支自动发现；
    • 任务创建时没有 team preamble；
    • 跟 EACN3 官方案例 001–003（多 Agent 协作的样板）对不齐。
  3. invite_agent 与 invited_agent_ids 用得不一致。 EACN3 的 admission threshold（confidence × reputation ≥ θ）会挡住新 Agent（reputation=0.5）对公开任务的竞标。invite_agent 绕过能力检查。 我们在 create_task 时用了 invited_agent_ids，但任务发出后的"增补邀请"没跑通。
  4. 不读 reputation。 Role 之间互相选任务不看对方 reputation、人手动 report 也不做（全靠 submit_result/reject_task 的自动上报）、Gru 的心跳摘要也没把信誉漂移作为信号暴露给人。结果：Role 反复失败时 EACN3 知道，但人不知道。
  5. 经济系统形同虚设。所有任务 budget=0，deposit 只为保险丝而设：
    • escrow / settlement 逻辑完全未走；
    • confirm_budget / bid_request_confirmation 这个天然的"Role 因价太低拒单"信号用不上；
    • 公开任务若 budget > 0，每 role 的 deposit 管理我们没做。
  6. _post_message_raw 发空 server_id / network_id。 确认：单机下无害（本地投递只看 agent_id），但未来跨节点转发（forward_push）会路由失败。两种选择：
    • A. 从目标 AgentCard 查出再填（一次 get_agent_card，便宜）。
    • B. 文档声明 MinionsOS 有意单节点部署。
  7. 事件状态三重存储。插件 state.ts::drainEvents 缓冲合成事件、MinionsOS pending.jsonl 做 ACK 补丁，EACN3 OfflineStore 存权威。每个事件短时间内存在三处，功能不重叠但状态机耦合。
  8. 不调用 eacn3_close_task。EACN3 的任务状态机有 completed 和 closed 两个终态；我们在 select_result 后停住，于是任务一直以"open but completed"存在。无害但长期堆积。
  9. 任务构造参数欠缺。create_task 接受 level / max_depth / max_concurrent_bidders，MinionsOS 全部使用默认。 我们本可以用它们从 role prompt 侧约束 review round 的并发评审数量或 subtask 深度。
  10. 不用 eacn3_discover_agents。EACN3 支持 Gossip → DHT → Bootstrap 三级发现。单节点下 list_agents 够用；联邦化后这个工具是入口。

其他小观察

  • 插件的 eacn3_await_events client-side cap 是 120 秒，但后端 422 拒绝 > 60，所以端到端上限依然是 60 秒。我们的 MOS Agent Pool 正确切片在 60 秒。
  • eacn3_list_open_tasks 后端最多 200 条/次；我们默认 50，够用。
  • OfflineStore 有 max_per_agent 和 ttl_seconds 限制 —— 长时间休眠的 role 可能丢消息。应在文档中说明。
  • eacn3_get_task_results 首次调用 会把任务状态推向 completed（side effect）。Role 做诊断读时要注意。

9. 优化建议（按性价比排序）

高价值 / 低成本 QUICK WIN

  1. 在 select_result 之后调 eacn3_close_task。改公共 EACN 技能即可。让任务市场保持干净。
  2. _post_message_raw 从 AgentCard 查出三层地址。一次 get_agent_card 查询，成本几乎为零。以后若联邦化无需再改。
  3. 在 mos_create_task 上设 max_depth=3 / max_concurrent_bidders=<N>，尤其是 review round / experiment，给我们一个协议原生的速率限制。
  4. 在公共 EACN 技能里文档化 get_task_results 的副作用。防止 Role 做诊断时误伤状态机。
  5. 在 Gru 心跳摘要里暴露 reputation。每 role 一次 eacn3_get_reputation。把隐式的信誉漂移变成可见信号。

中等价值 / 中等成本 WORTHWHILE

  1. 在 MinionsOS 项目里启用 eacn3_team_setup。 branches/<role>/ 已经就是 EACN3 团队所期望的布局。实施路径：
    • project_create 注册完所有 role 后，用完整 role 集调 eacn3_team_setup；
    • EACN3 会自动往任务里注入 team preamble，Role 之间的分支认知免维护；
    • 对齐官方案例 001–003 的团队范式。
  2. 把 eacn3_next 包一层 mos_next 作为可选调度器。主循环仍是 mos_await_events；想要 EACN3 urgency + idle prompt 的 role 按需用。能省大量 prompt token。
  3. Review 轮里使用 invite_agent。Reviewer 主会话事后想拉 Ethics / Expert 参与复审时，用 invite_agent 绕过 admission threshold 是合法途径。

低价值 / 高成本 LATER

  1. 真实 budget + confirm_budget 驱动议价。EACN3 原设计故事，但单租户封闭项目收益低，建议推迟到多租户。
  2. 采用 eacn3_discover_agents。MinionsOS 未来跨项目联邦化时的入口，当前单项目 list_agents 已够。
  3. 停止维护 pending.jsonl，完全依赖 OfflineStore TTL + 手动 replay。只有当三重存储真的成为调试负担时再动手。

建议的第一步

把上面 1–5 条做成一个小 PR：三行代码 + 文档更新。team_setup（第 6 条）值得单独一个 PR，因为它需要在 project_create 和 register_role 之间协调，并更新 common SYSTEM.md 的 team preamble 约定。
