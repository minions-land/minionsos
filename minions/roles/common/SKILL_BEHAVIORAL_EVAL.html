49 个 Skill，到底有几个真有用？

我们让 Haiku 在"有 skill"和"没 skill"两种条件下回答同一道情景题， 再让 Codex 蒙着眼裁哪份回答更好。这是 MinionsOS skill 库第一次被 系统地从"文档质量"切到"实际行为效果"上检验——以及结果为什么 比想象中刺眼。

评审 Opus 4.7（Meta 层）+ Codex GPT-5.5（盲判）+ Haiku 4.5（执行） · 阅读 约 12 分钟 · 2026-05-14

I一份没人读得懂的报告

问题不是 skill 写得好不好。问题是它真的让 agent 做得更好吗？

在跑这一轮之前，我们已经做过一次评估：让五个 Sonnet 评审员把每个 skill 文件按 SSL 四节模板审一遍——summary 清不清楚、trigger 具不具体、Procedure 能不能照做、 Pitfalls 是否抓到真失败。那一轮结论很漂亮：49 个 KEEP，9 个微调，0 个重写。 但那只回答了一个问题，文档写得好不好。它没回答另一个完全不同的问题： skill 真的让 agent 做得更好吗？

这两个问题的答案可以完全不同。一个写得平庸的 skill 可能恰好教会 agent 它本来不会 做的事；一个写得漂亮的 skill 可能只是把模型本来就会的事又说了一遍——后者就是 "花瓶 skill"，文档评估抓不到。

所以这一轮做的事完全不一样。我们不审 skill 写得怎样，我们看 skill 实际能不能改 Haiku 的行为。49 个 skill，每个都给一道情景题，让两个 Haiku agent 同时回答—— 一个在 prompt 里塞进了完整的 skill 文本，另一个没有。然后把两份回答以随机标签 （RED / BLUE）扔给 Codex，让它不知道哪份是 with-skill，只按决策质量裁。

这份报告有两个大发现。第一个是好消息：MinionsOS 的 skill 库整体在工作—— 65% 的 skill 在行为层面有可测量的正向效果。第二个是更重要的：哪些 skill 在工作，怎么工作的，跟我们直觉不太一样。

II实验设计

为什么用 Haiku 当执行者，而不是 Sonnet。

评估 skill 的真实效果有一个核心难题：不能用一个本来就什么都会的模型去测。 给 Sonnet 加一份"think before coding"的 skill，它本来就会 think before coding， 差异淹没在噪音里——你测不出来。Haiku 不一样，它会跳步、会走捷径、会忘掉自查， 它对 skill 的反应才是可观察信号。

这也吻合 production 现实：MinionsOS 里 role wake 默认用 Codex 或 Haiku 这种快模型， 不会动用前沿模型。skill 真正要服务的就是这些 agent。所以让 Haiku 来当被试，比让 Sonnet 来当被试，离实际部署场景更近。

III结论先看

49 个 skill 的真实分布——以及为什么"一半看起来没用"其实不是坏消息。

把所有 61 道 probe 的判决合到一起，得到下面这张表。每个 skill 进入其中一类， 依据是 Codex 的 winner、skill_effect_estimate 字段，加上我对回答内容的对照 （比如"BLUE 凭空发明了一个不存在的工具名"算"阻止真错误"，无论 Codex 是否标 "skill helped"）。

分类数量含义
阻止真错误 18 没有这个 skill，Haiku 会在决策层面犯错——发明假工具名、绕过 MCP 接口、 擅自实现没人要求的事、违反 Pass A 隔离规则等。
校准回答 14 不会改变最终决定，但让回答更具体、更有依据、更不会滑向"crank mode"或 过度宽泛的 caveat。

过度执行（已修复） 1 coding-methodology 第一版："改超过 20 行就自动触发 Phase 3" 这条规则在用户没要求清理时也会触发。修复后行为正确。

用一句话概括：三分之二的 skill 在行为层面有正向效果。其中最值钱的是 "阻止真错误"那 18 个——它们不是让回答更漂亮，而是把一个会失败的决策翻成一个能 成功的决策。

Haiku 在不确定时会发明工具名。skill 充当的是词汇表锚点。 Codex 在 11 次判决里独立得出的相同观察

第二大类（校准）的价值更隐蔽：两份回答都对，但 with-skill 版本的论据更紧。 比如 first-principles：两个 Haiku 都对"用 Adam 因为大家都用"这句话 push back，但 with-skill 版本停在"Adam 是合理 default、需要时再换"这种 calibrated 立场，baseline 版本滑得更远，几乎在要求做完整 sweep——后者更接近 skill 警告的 "crank mode"。

第三大类（持平）才是这次评估最容易被误读的部分。下文第六节专门讨论它。

IV三个最值钱的子库

数据指向的优先级——以及它跟我们直觉的偏差。

EACN3 cluster：阻止 API 幻觉

最让我意外的发现：当 Haiku 没有 skill 加载时，它会发明根本不存在的 EACN3 工具名。 这不是猜测——Codex 在多个 probe 里把这件事一一引出来。

baseline 编出来的工具名实际正确的工具
eacn3_query_agentseacn3_discover_agents
eacn3_read_taskeacn3_get_task_status
eacn3_get_healtheacn3_health
eacn3_retrieve_task（"待获取后才解锁"）FSM 里没有这步

所有 13 个 EACN3 cluster skill 测下来，每一个都要么阻止真错误、要么校准回答。 没有一个跟 baseline 持平。这是整个 skill 库里 ROI 最高的子库——对 Haiku 这种没有把 EACN3 工具表"烧进权重"的执行者来说，cluster skill 不是装饰，而是必需品。

工作流编排 skill：阻止过度执行

第二个清晰的模式：当"正确答案 = 先停下来 / 先 scope / 别动手"时，baseline Haiku 倾向于 立刻动手。with-skill 版本则知道刹车。下面这些都是真实失败被阻止的例子：

Skillbaseline 的错误
feature-implementation用户只说"加 --json flag"，baseline 直接 commit 了一个版本（连 schema 都猜的）
apply-revisions跳过 revision checklist，直接往 paper 上改
paper-work-boundaries把任务派给"Coder subagent"而不是 paper-figure-python，破坏 boundary map
aspect-review计划读 round-1 reviewer 报告（违反 Pass A 隔离）
figure-spec直接写 TikZ/pgfplots，跳过 JSON spec → SVG renderer 流程
archive-execution把 9 个 run 全归档（skill 要求只归档recurring pattern）
track-run擅自 kill 一个慢但健康的训练（应该先报给 requester）

推理纪律 skill：校准价值

first-principles、dialectics、citation-authenticity-audit、 coding-methodology（修复后版本）这一组，提供的不是"翻转决策"的价值，而是 "提升决策论据质量"的价值。两份回答都对，但 with-skill 版本能给出更准的采样权重、 更清晰的 thesis-antithesis-synthesis 结构、更校准的 push back。

这一组很容易被低估。它们不会救你不犯错，但能让你的回答经得起更严苛的 review。 在科学发表场景里——比如 Ethics 审计 80 个引用、Expert 推 first-principles 论证——这种"地基整齐"的价值是真金白银。

VHaiku 自己的盲点

三个反复出现的失败模式——这些影响 skill 该怎么写。

跑完 122 个 Haiku 调用之后，有三个模式在数据里反复出现，强烈到值得单独命名。 它们不是"Haiku 是劣质模型"，而是"Haiku 在没有 scaffold 时的默认倾向"—— skill 设计应该围着这些倾向打。

1. 不确定时凭空发明工具名

这是 Codex 高置信度判出来"with-skill 受益"最大的来源。Haiku 在不知道一个 API 的精确名字时，不会说"我不确定"，而是会编一个听起来合理的名字 （eacn3_query_agents、eacn3_get_health），并把它放到 回答里像真有这个工具一样。skill 在这里起的作用是词汇表锚点——把正确的 工具名以高频率出现在 Haiku 的 context 里。

2. 上下文不足时就退到"我需要用户告诉我 project port"

很多 baseline 回答开头是"No active projects, what's the port?"——在真正模糊的 场景里这是好习惯，但当 prompt 已经给出足够信息时这变成失误。skill 给 Haiku 足够的 scaffolding 让它有信心直接动手，而不是退到"问用户"这条 fallback。

3. 任务开放时倾向于过度执行

"Add a --json flag" → baseline 立刻实现了。
"Run the autoresearch skill" → baseline 试图找一个不存在的 skill 文件。
"Submit before deadline" → baseline 跳过 citation 审计直接发。

带有显式澄清门的 skill（"先问，再写"）反复阻止这种 silent scope guessing。 这就是为什么 feature-implementation、apply-revisions、 feature-intake 等都进了"阻止真错误"那一类。

设计含义 给 Haiku 写 skill，最关键的不是把方法论解释清楚，而是把正确的工具名和 正确的工作流名以足够高的频率重复在 skill 文本里。Haiku 不会因为读懂了 skill 才用对工具——它会因为正确的工具名在它的 context 里，所以倾向于用它而 不是发明新名字。这是为什么 EACN3 cluster skill 全部命中正向效果：每个文件 都密集地、反复地写出真实工具名，等于在抗幻觉。

VI"看起来没用"的 14 个

这不一定是失败——可能只是问题问得不够难。

14 个 skill 在标准 probe 下跟 baseline 持平。第一反应可能是"那这些 skill 没用， 可以删了"。但有一段补充测试改变了这个判断。

第一轮 12 个 probe 跑完后，有 4 个 skill 显示"无效"。我们做了一轮"难场景重测"—— 把每个 skill 的 probe 改成更模糊、更接近边缘情况的版本，再跑一次。结果如下：

Skill标准场景难场景真实定性
bounded-repair-loop 持平 持平 真低杠杆——Haiku 自己也知道何时停手
simulate-reviewer-instance 持平 持平 真低杠杆——3-vs-4 的判断对 Haiku 显而易见
abstract-writing 持平 阻止真错误 之前误判——baseline 在难场景下会直接 expand 第一段，with-skill 知道要重新结构化
triage-request 持平 阻止真错误 之前误判——baseline 在难场景下把决策上交给 Expert，with-skill 自己做 queue + dependency

一半的"无效"判定其实是错的——只是 probe 设计太宽松。这是这次评估方法学上最大的 教训：单一标准场景可能高估也可能低估 skill 的价值。

所以剩下 10 个"无效"skill 的真实地位仍然未知。它们可能是真低杠杆，也可能是 "在更困难的边缘情境下其实有用、只是这一轮没测到"。基于现有证据，不建议删除。 但下一轮回归测试值得设计更难的 probe。

另一个维度：稳定性 vs 决策翻转

一个值得正视的可能性：某些"无效"skill 的真实价值不是"翻转 Haiku 的决策"，而是 "稳定Haiku 的决策"。即在标准场景下 Haiku 自己也对，但温度 / 模型版本 / 随机性可能会让它偶尔走偏；skill 的存在让它走偏的概率下降到接近零。这种稳定性 价值在单次 A/B 里测不出来，需要重复多次跑同一个 probe 才能看到。

这次评估每个 probe 只跑一次，所以无法分离"决策翻转"和"决策稳定"两种贡献。 把它当作下一轮的开放问题。

VII下一步

基于这份评估，三个具体决定。

1. 全部 49 个 skill 都保留

没有任何一个 skill 在修复 coding-methodology 之后表现出净负效果。 14 个"看起来没用"的 skill 暂时保留——直到我们有更难的 probe 能区分"真低杠杆" 和"暂时没测到价值"。

2. EACN3 cluster 视为最高优先级 skill

13 个 EACN3 skill 全部命中。这是整个库 ROI 最高的子库，因为它直接抗幻觉： Haiku 在没有这些 skill 时会发明工具名。每个使用 EACN3 的 Role wake 应该确保这一类 skill 全部加载。这一点已经通过 list_skills() 的自动发现机制保证了。

3. 推理纪律 skill 不要被低估

first-principles、dialectics、citation-authenticity-audit 这一组提供的是校准价值，不是错误阻止价值。它们让 Expert / Ethics 的回答能 经得起更深的 review——这在科学发表场景里直接转化为论文质量。不要因为它们"没翻转 决策"就降低优先级。

方法学的两个开放问题

第一个问题：本次每个 probe 只跑一次。若同一道 probe 跑 5 次取众数，"决策稳定性" 的贡献会显现出来。

第二个问题：盲判设置依赖 Codex 不识别 skill 痕迹。但 skill 的 vocabulary（"Phase 3"、 "六阶梯"等）有时会在 with-skill 回答里直接出现，让盲判其实不够盲。下一轮可以在 判前做一次"术语漂白"——把 skill-specific 名词替换成中性术语，再喂给 Codex， 看 Codex 还是不是判 with-skill 占优。

给以后做这件事的人

如果以后想跑同样的回归测试，方法是稳定的：49 个 skill × 标准 probe + 难 probe， 每个 probe 4 个 Haiku Agent + 1 个 Codex 判，分批 spawn 控制 context cost。 全程约 3 小时 wall clock，~17M token 总开销。本次完整 raw transcript 保留在 minions/roles/common/_evals/。
