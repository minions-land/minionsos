# Skill — Figure Spec

A structured specification format for formal architecture / workflow / pipeline figures — deterministic, editable, and review-friendly.

## Core move

Before generating a diagram, write a FigureSpec: an explicit JSON-like description of canvas, nodes, edges, and groups. The spec is the source of truth; the rendered SVG/PDF is built from it and can be regenerated byte-stable from the same spec.

## Procedure

1. **Decide when to spec.** Use for: architecture diagrams (layered / hub-and-spoke / multi-plane), workflow pipelines, audit/flow-control cascades, any figure whose structure is semantically meaningful. Do **not** use for data plots (→ `academic-plotting`) or informal illustrations.
2. **Draft the spec** with four blocks:
   - `canvas`: `{width, height}` — single-col ~500×350, two-col ~900×500, tall topology ~700×700.
   - `nodes`: list of `{id, label, x, y, shape, fill, stroke}` — positions are explicit; no auto-layout magic.
   - `edges`: list of `{source, target, style, label}` — `style` ∈ `solid` (data flow), `dashed` (control / feedback), `gray` (secondary).
   - `groups`: optional list of `{label, node_ids, fill, stroke}` for named regions.
3. **Name labels exactly as they appear in the paper.** The spec is also a glossary check — label drift between figure and prose is a common review smell.
4. **Render deterministically.** Use a JSON → SVG renderer (e.g. `tools/figure_renderer.py` if present in `workspace/`). Commit both the spec and the rendered SVG/PDF.
5. **Iterate on the spec, not the SVG.** Never hand-edit the rendered SVG for structural changes; edit the spec and re-render. Hand-editing only for final visual polish that won't regenerate.
6. **Include a provenance comment** in the spec file: which section of the paper this figure serves, which claim it supports, and the date of last regeneration.

## When to invoke

- Any formal architecture / method-overview / pipeline figure for a venue submission.
- When a reviewer asks "how does X connect to Y?" and the answer needs a canonical diagram.

## Pitfalls

- Letting a diagram tool auto-layout for you. Spatial arrangement carries meaning; specify it.
- Drifting labels between figure and prose. Lock label text in the spec.
- Hand-editing SVG to patch the spec's mistakes — breaks regeneration and future edits.

## Output habit

Spec and rendered figure live side-by-side under `workspace/paper/figures/` (e.g. `fig_overview.json` + `fig_overview.pdf`). Spec provenance comment cites the supporting section per root §9.
