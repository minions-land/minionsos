评测方法论：MetaHarness 行为 A/B 框架

MetaHarness 是一个行为级 A/B 评测框架，测试的是一个 skill 是否真的改变了 agent 的行为，而不是文档写得好不好。它使用两阶段流水线：召回测试（skill 能不能被找到？）+ 行为对比（加载后有没有帮助？）。

设计哲学

核心问题：把这段 skill 文本注入 agent 的上下文后，它的决策质量是否可测量地优于没有 skill 的同一个 agent？

为什么不直接读 skill？一个 skill 可以写得很漂亮但行为效果为零。反过来，一个粗糙但决策边界正确的 skill 可以防止灾难性错误。MetaHarness 测量的是效果，不是形式。

目标模型：Haiku（小、快、便宜）。这是刻意的选择——Sonnet/Opus 本身已经能做到 skill 所规定的大部分事情，差异消失在噪声中。Skill 的设计目标是提升低成本执行器的下限。

两阶段流水线

阶段 0：SSL 召回测试 阶段 1：行为 A/B 对比 ┌───────────────────────────────┐ ┌───────────────────────────────┐ │ 输入：所有 skill 的 summary │ │ 对每个 probe： │ │ + 场景描述 │ │ │ │ │ │ Haiku A（注入 skill 文本） │ │ 问："为这个场景选 top-3 skill" │ │ Haiku B（无 skill，基线） │ │ │ │ │ │ 指标：Recall@3, MRR │ 通过 │ 两者收到完全相同的场景 │ │ │ ──────▶ │ 唯一区别：有无 skill 文本 │ │ 如果 Recall@3 = 0： │ │ │ │ 停止。先修 summary。 │ │ Codex GPT-5.5 盲评 │ │ │ │ （随机 RED/BLUE 标签） │ └───────────────────────────────┘ └───────────────────────────────┘

阶段 0：SSL 召回测试

测什么：给定一个场景，agent 的 skill 选择步骤能否把目标 skill 放进 top-K？

为什么重要：一个找不到的 skill 就帮不上忙。如果 summary 太泛、用了内部术语、或者和兄弟 skill 重叠，agent 会选错——Stage 1 的结果就变成假阳性。

指标阈值动作
Recall@3 = 1, MRR ≥ 0.5健康进入 Stage 1
Recall@3 = 1, MRR < 0.5边缘标记干扰项；谨慎进入 Stage 1
Recall@3 = 0失败停止。先重写 summary 再做 Stage 1

成本：~60 次 Haiku 调用 × ~3K tokens = ~180K tokens。约 5 分钟。便宜的保险——避免在不可达的 skill 上浪费 Stage 1 预算。

阶段 1：行为 A/B 对比

协议：

  1. 设计 probe——≤300 字的场景提示，有明确正确答案。每个 probe 包含：具体角色、具体场景（文件路径、数字、状态值）、具体问题（"你的下一步行动是什么？"）。
  2. 生成 Haiku 对——每个 probe 两个 agent，除了一个注入了 skill 文本外完全相同。后台运行，≤200 字回复。
  3. Codex 盲评——GPT-5.5 以 reasoning_effort: low 用随机 RED/BLUE 标签评判。无法识别哪个回复有 skill。输出严格 JSON：{winner, reasoning, skill_effect_estimate, confidence}。
  4. 分桶——每个 skill 被归入四种结果之一。
分桶含义示例
防止真实错误没有 skill 时，基线做出错误决策编造不存在的 API 名、过度执行、调用禁止的工具
校准回复两者都答对；有 skill 的版本论证更充分引用权威来源、使用结构化格式
与基线持平无可测量差异probe 太简单，或 skill 与训练冗余
过度干预有 skill 的版本比基线更差自动触发用户未请求的工作

Probe 设计原则

标准 probe 测试 skill 应该触发的常见场景。

困难 probe 测试边界情况——skill 的价值在于克制（不行动），而朴素 agent 会行动。例如：

  • "测试在第 2 次迭代后通过了，但你改了 47 行，最小修复只需 5 行。"
  • "用户说的是'直接实现'，不是'实现并清理'。"
  • "审稿人 1 = 强接受；审稿人 2 = 拒绝。继续还是停止？"

反模式："你醒来发现有事件"（太模糊）、"帮用户处理论文"（没有可判定的问题）、正确答案是"问一个澄清问题"的 probe（A 和 B 都会问）。

模块 1：Skills 技能（Think-Then-Act）

Think-Then-Act 是什么

一个结构化规划技能，提供五个可组合的"思维姿态"，让自主 agent 在行动前先推理。Agent 自己决定用哪些姿态、用几个、什么顺序。思考完毕后，所有执行都派发给子 agent——永远不自己动手。

姿态何时使用何时跳过
未陈述前提任务有未说明的约束规格具体且可验证
第一性原理"大家都这么做"是最强论据时惯例本身就是承重的
辩证综合两种方案冲突没有真正的对立面
目标设定派发前需要验收标准二元检查就够了
计划持久化多步工作跨越上下文重置单步任务

核心设计原则：上下文分离——思考上下文保持干净用于协调；执行上下文是一次性的。这防止了上下文膨胀在多任务会话中降低规划质量。

证据 1：MetaHarness 行为 A/B（全库运行）

设置：49 个 skill × 1-2 个 probe，Haiku 执行器，Codex 盲评

TTA 子姿态测试结果：

子姿态分桶细节
first-principles校准回复保持校准；不会滑入偏执模式
dialectical-synthesis校准回复正题/反题/合题结构，带条件性解决
coding-methodology防止真实错误没有它时，基线会过度执行

总体发现：32/49 个 skill（65%）有可测量的正面行为效果。18 个"防止真实错误"的案例是承重核心。

Haiku 的三种失败模式（skill 可以纠正的）：

  1. 不确定时编造工具名——skill 充当词汇锚点
  2. 上下文不足时默认"问用户"——skill 提供脚手架让 agent 自主行动
  3. 开放式任务上过度执行——带澄清门控的 skill 防止静默猜测范围

Think-then-act 直接解决失败模式 #2 和 #3。

证据 2：竞品对比评测（2026-05-17）

设置：3 个 probe × 4 个变体（TTA v3、brainstorming、writing-plans、ultrathink）

Probe类型TTA v3 结果胜者
P1具体规格败——在简单任务上搞仪式brainstorming
P2模糊架构败——给离线角色发消息而不是查代码库writing-plans
P3多角色启动败——在交接时软化了阈值writing-plans

ultrathink 三局全败——没有结构的纯算力不管用。

修复措施（v3 → v6）：

修复内容解决
(a) 跳过触发器"规格具体时直接跳过"P1
(b) 证据优先先在代码库上跑 codex/Task，再发 EACN 消息P2
(c) 逐字传递目标"TBD"、"合理的"都算计划失败P3
(d) 具体示例纯文字规则在 v5 无效；加了具体失败模式示例后生效P2

修复后结果：P2 重测 → Haiku 正确地用 codex 查了具体文件（role.py、whitelist.py、eacn3.db），而不是给离线 Writer 发消息。

证据 3：自评功能移除（负面结果）

设置：5 个 probe 的 Haiku A/B（4 难 + 1 简单），测试"推荐自我评估"功能

结果：2/5 胜，且有混淆因素。功能已移除。

失败模式：

  • Haiku 写的是防御性复述而非诚实反思（3/5 被 Codex 标记为"无新洞察"）
  • 在简单任务上，反思提示推动 agent 过度执行（P5：自动提交超出范围）
  • Sonnet/Opus 天然会自我评估——显式提示对大模型冗余，对小模型有害

意义：证明 skill 设计过程有移除机制，不只是堆积。功能基于证据被测试和拒绝。

证据 4：姿态 4 漂移修复（2026-05-19）

观察：在一个投票机制设计问题上，姿态 4（目标设定）坍缩成了"工程性结论"，而不是产出规范的 5 要素 Goal 块。

漂移模式：综合以"所以我们应该建 X"结尾 → agent 把目标设定当成"展开推荐" → 产出 yes/no 判断，没有传感器、阈值或停止规则。

修复：显式失败模式提示："如果你的输出读起来像执行摘要，你跳过了姿态 4。"

意义：Skill 需要反漂移机制。自然语言流（综合→结论）会绕过结构化输出要求，除非显式守卫。

建议的优先级（先做哪些）

  1. 金标 — PaperBench（8316 rubric leaves，端到端复现 ICML 2024 paper）：能跑通就是 MinionsOS "全栈实力"的最强 evidence。
  2. 核心战场 — SWE-bench / SWE-bench Pro / MLE-Bench / MLAgentBench：Coder + experiment_scheduler + Codex 是已有最成熟的链路，先跑这一组建立 e2e 数字。
  3. 差异化 — ClaimCheck + AI Detection + Breaking the Reviewer：测 Ethics + mos_review_run 双重审查的差异化优势（v.s. 单代理 review）。
  4. 记忆/长程 — RE-Bench + SWE-bench Pro：测 Reel L0 + Draft L1 + Book L2 的长程协作价值，这是与所有竞品最大的架构差。
  5. 角色演化 — MLR-Bench + FrontierScience + HLE：测 mos_role_evolve_evaluate 的 split/merge/dismiss 在多领域开放任务上的效用。

使用须知（不是所有 benchmark 都该等价对待）

  • 2026 年生成的 benchmark 编号有错位风险：原 README 多个条目用 arXiv:2601.xxxxx 等占位 ID，正式开跑前需用 GitHub repo / DOI 实际确认；从 README 中复制的 arXiv 链接需逐一核验。
  • Saturation 警告：GAIA / SimpleQA / SWE-bench 等 2024 之前的 benchmark 在 2026 年可能已被前沿模型 saturate；它们应作为 sanity check，而不是 headline result。
  • License + 评分 API：很多 benchmark 需 OpenAI judge / GPT-4 judge；列入预算。可用本地 Codex GPT-5.5 替代以节约。
  • 对照对手：每跑一个 benchmark 都至少再跑一次 baseline（同模型裸 Claude Code 或 single-agent loop），否则数字无解读价值。
